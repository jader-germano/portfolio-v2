# Data Model

## Scope

This model documents the storage boundary for the currentNest.js backend/BFF
lane and the future Java + Quarkus replica.
The current application remains a singleNest.js 14 runtime for auth, route
handlers, orchestration and Supabase-facing contracts.
The model below prepares the database contract without prematurely locking the
resume-upload implementation before discovery closes.

## Architectural Direction

- Current frontend: separate portfolio shell consuming this backend/BFF.
- Current backend lane:Nest.js App Router handling auth, protected APIs,
  orchestration and Supabase contracts.
- Current auth/session layer: `next-auth` for web session management, Supabase
  Auth for credential validation, and `profiles` for role metadata.
- Future final-epic lane: a separate Java + Quarkus service replicating the
  same contracts after the VPS deployment is stabilized.

## Supabase Boundary

Supabase is the current managed boundary for:

- Auth identities and provider metadata
- Postgres persistence
- Storage/object references when uploads become concrete

The application layer must document exactly which semantics live in Supabase and
which rules live in the backend so a future Java + Quarkus lane can mirror the
behavior without changing domain meaning.

## Scalability Expectations

TheNest.js backend/BFF must be designed with business-level growth in mind:

- support horizontal scale with stateless web nodes
- keep session state and persistence externalized
- move long-running or bursty work behind async/job boundaries
- preserve idempotency for retries and duplicate submissions
- expose enough metrics and audit data to diagnose saturation

## Logical Domains

### Identity and Access

#### `auth.users`

Managed by Supabase Auth.

| Column | Type | Notes |
| :--- | :--- | :--- |
| `id` | `uuid` | Primary key generated by Supabase Auth |
| `email` | `text` | Unique login identifier |
| `encrypted_password` | managed | Stored by Supabase Auth |
| `raw_user_meta_data` | `jsonb` | Provider/user profile payload |
| `created_at` | `timestamptz` | Auth creation timestamp |

#### `public.profiles`

Current role extension table already documented in `docs/rbac_setup.sql`.

| Column | Type | Notes |
| :--- | :--- | :--- |
| `id` | `uuid` | PK, FK to `auth.users.id` |
| `email` | `text` | Duplicate lookup field for ops and policies |
| `role` | `app_role` | `master_admin`, `admin`, `user` |
| `display_name` | `text` | Optional friendly label for header/session UI |
| `created_at` | `timestamptz` | Profile creation timestamp |
| `last_login_at` | `timestamptz` | Optional operational audit field |

#### `public.legal_acceptance_log`

Tracks acceptance of Terms/Privacy versions for authenticated users.

| Column | Type | Notes |
| :--- | :--- | :--- |
| `id` | `uuid` | Primary key |
| `user_id` | `uuid` | FK to `public.profiles.id` |
| `document_type` | `text` | `terms_of_service` or `privacy_policy` |
| `document_version` | `text` | Semantic or dated version string |
| `accepted_at` | `timestamptz` | Acceptance timestamp |
| `source_provider` | `text` | `github`, `google`, `credentials` |
| `ip_hash` | `text` | Optional hashed client IP for audit only |
| `user_agent_hash` | `text` | Optional hashed user agent |

#### `public.access_audit_events`

Minimal access/audit registry for protected content.

| Column | Type | Notes |
| :--- | :--- | :--- |
| `id` | `uuid` | Primary key |
| `user_id` | `uuid` | Nullable FK to `public.profiles.id` |
| `event_type` | `text` | `sign_in`, `sign_out`, `session_expired`, `route_access`, `download_access` |
| `route` | `text` | Requested page or asset route |
| `provider` | `text` | Auth provider used |
| `metadata` | `jsonb` | Sanitized operational context |
| `created_at` | `timestamptz` | Event timestamp |

### Portfolio Instantiation and Uploads

Current status: **requirements collection / discovery phase**.
The tables below represent the likely contract direction, not a committed final
parser implementation yet.

#### `public.portfolio_uploads`

Stores upload metadata for resumes and source files.

| Column | Type | Notes |
| :--- | :--- | :--- |
| `id` | `uuid` | Primary key |
| `user_id` | `uuid` | Nullable FK to `public.profiles.id` |
| `storage_path` | `text` | Bucket/object path |
| `original_filename` | `text` | User-provided filename |
| `mime_type` | `text` | `application/pdf`, `text/plain`, etc. |
| `size_bytes` | `bigint` | File size |
| `sha256` | `text` | Deduplication and integrity check |
| `status` | `text` | `uploaded`, `processing`, `parsed`, `failed` |
| `created_at` | `timestamptz` | Upload timestamp |

#### `public.portfolio_generations`

Tracks generation requests and lifecycle.

| Column | Type | Notes |
| :--- | :--- | :--- |
| `id` | `uuid` | Primary key |
| `user_id` | `uuid` | Nullable FK to `public.profiles.id` |
| `upload_id` | `uuid` | FK to `public.portfolio_uploads.id` |
| `status` | `text` | `queued`, `running`, `completed`, `failed` |
| `llm_model` | `text` | Example: `deepseek-r1:7b` |
| `prompt_version` | `text` | Prompt/template version |
| `error_message` | `text` | Nullable failure detail |
| `started_at` | `timestamptz` | Processing start |
| `finished_at` | `timestamptz` | Processing end |
| `created_at` | `timestamptz` | Request timestamp |

#### `public.generated_portfolios`

Persisted normalized output from the generation pipeline.

| Column | Type | Notes |
| :--- | :--- | :--- |
| `id` | `uuid` | Primary key |
| `generation_id` | `uuid` | FK to `public.portfolio_generations.id` |
| `slug` | `text` | Public portfolio slug |
| `payload` | `jsonb` | Structured portfolio output |
| `render_version` | `text` | Renderer/version stamp |
| `published_at` | `timestamptz` | Nullable publication timestamp |
| `created_at` | `timestamptz` | Persistence timestamp |

### Private Knowledge and Docs

Indexes protected docs, PDFs, and private assets.

| Column | Type | Notes |
| :--- | :--- | :--- |
| `id` | `uuid` | Primary key |
| `slug` | `text` | Stable lookup identifier |
| `title` | `text` | Display title |

## Storage Boundaries for the Future Split

### Frontend responsibilities

- Public portfolio rendering
- Login shell, legal pages and session UX
- Upload intake UX and discovery feedback
- Consumption of protected APIs and dashboard data

###Nest.js backend/BFF responsibilities

- Auth/session core
- Supabase contract ownership
- Persistence for uploads, generation jobs, and private documents
- Workflow/job orchestration
- Audit/event ingestion
- API contracts consumed by the portfolio frontend and future mobile lane

### Future Java + Quarkus replica responsibilities

- Implement the same API contract and storage semantics
- Reuse the documented Supabase boundary without reinterpretation
- Pass contract and parity tests against theNest.js lane
- Enter only in the final post-deploy replica epic

## Recommended Next Extraction Order

1. Keep theNest.js lane stateless and horizontally scalable.
2. Document Supabase deeply enough to support a later Java + Quarkus replica.
3. Treat resume upload as discovery until the intake requirements are frozen.
4. Move upload/generation/document access to database-backed endpoints only after discovery closure.
5. Extract the canonical API contract that the future Java + Quarkus lane must replicate.
